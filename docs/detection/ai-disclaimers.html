Deployment Strategies

Last updated: March 8, 2026 · 12 min read

Arcline supports multiple deployment strategies to match your team's risk tolerance and operational requirements. This guide covers the most common approaches and when to use each one.

Blue-Green Deployment

Blue-green deployments maintain two identical production environments. At any time, only one environment serves live traffic while the other remains idle or runs the new version.

  • Zero-downtime releases with instant rollback capability
  • Full environment validation before switching traffic
  • Higher infrastructure cost due to duplicate environments
arcline deploy --strategy blue-green \
  --image registry.arcline.io/app:v2.4.1 \
  --health-check /api/health \
  --switch-after 300s

Canary Releases

Canary deployments gradually shift traffic from the old version to the new one, monitoring key metrics at each step. If anomalies are detected, the rollout is automatically paused or reverted.

Recommendation: Use canary deployments for services with high traffic volume where you need statistical confidence before full rollout. 
arcline deploy --strategy canary \
  --image registry.arcline.io/app:v2.4.1 \
  --steps 10,25,50,100 \
  --interval 600s \
  --metric-threshold error_rate<0.01

Rolling Updates

Rolling updates replace instances one at a time (or in small batches), ensuring that a minimum number of healthy instances are always available. This is the default strategy for most Arcline services.

  • Lower resource overhead compared to blue-green
  • Gradual rollout with automatic health checking
  • Rollback requires redeployment of the previous version

Feature Flags

For high-risk changes, combine deployment strategies with feature flags. Deploy the code first, then enable the feature incrementally using Arcline's built-in flag system:

arcline flags set new-billing-engine \
  --rollout-percentage 5 \
  --target-group beta-testers

This approach decouples deployment from feature release, giving your team fine-grained control over who sees new functionality.

Choosing a Strategy

The right strategy depends on several factors:

  • Service criticality: Use blue-green for payment and auth services
  • Traffic volume: Canary works best with high-traffic endpoints
  • Team confidence: Rolling updates for well-tested, low-risk changes
  • Infrastructure budget: Blue-green requires 2x resources during deployment
